# AGENTS.md

## 0. 最重要原則（他のすべてに優先）
- 指示されていない作業をしない。「ついでに」の改善・リファクタ禁止
- 不明点・二義性があれば実装せず停止して質問する
- 自己申告の「動作確認しました」は無効。検証コマンドの実際の出力のみが証拠

## 1. 絶対禁止（MUST NOT）
- 依頼範囲外のファイル変更
- public API / 公開シグネチャの変更（Modの場合: 他Modから参照されうるクラス・メソッド全て）
- 新規依存の追加（必要なら停止して提案のみ）
- テストの削除・スキップ・アサーション弱体化による「通過」
- catch して log のみで握りつぶす例外処理 (必ずとは限らないが、基本的に禁止)
- TODO/FIXMEコメントで未完成のまま完了報告
- ハードコードされた秘密情報（APIキー、パスワード、トークン）
- `git push --force`, `git reset --hard`, 破壊的なDB操作

## 2. 検証（MUST・完了の定義）
以下すべてが通るまで「完了」と報告してはならない:
1. ビルド: `<プロジェクト固有: ./gradlew build / npm run build>`
2. lint: `<eslint / phpcs / checkstyle>`
3. テスト: `<./gradlew test / npm test / phpunit>`
4. 失敗時: 修正して再実行。3回失敗したら停止し、エラー全文を添えて報告

## 3. 作業単位とフェーズ
- 1タスク = 1機能 or 1ファイル。複合依頼は分割を提案する
- 新規実装は2フェーズ制:
  - Phase 1: ファイル構成・シグネチャ・責務のみ提示 → 承認待ち
  - Phase 2: 承認された設計から逸脱せず実装。逸脱が必要なら停止
- 各ファイル実装の直前に、本ファイルのセクション1,2,4を再確認すること

## 4. 構造ルール（数値で拘束）
- 1ファイル300行以内。超過見込みなら実装前に分割案を提示して停止
- 1メソッド/関数40行以内、ネスト3段以内
- 1ファイル1責務。「Utils」「Helper」への追記より新規クラスを優先提案

## 5. 例外処理（許可リスト方式）
try/catchの許可箇所:
- 外部I/O境界: ネットワーク、ファイル、DB、外部API
- エントリポイント最上位（main, リクエストハンドラ）
- 明示的に回復処理が存在する場所（リトライ、フォールバック）
それ以外は例外を伝播させる。

// GOOD: 境界でcatchし、文脈を付けて再送出
if (httpCLient == null) return null;
try {
    return httpClient.send(req);
} catch (IOException e) {
    throw new FetchException("API fetch failed: " + url, e);
}
// BAD: ロジック内で握りつぶし
try {
    if (data == null) return; // try外でよいものはtryに入れない
    process(data);
} catch (Exception e) {
    log.error(e); // 呼び出し元は失敗を知れない
}

## 6. コーディング規約
- （プロジェクト固有の命名・フォーマット規約をGOOD/BADペアで列挙）
- コメントは「なぜ」のみ。「何をしているか」のコメント禁止
- null: Java→Optional、TS→strictNullChecks前提、PHP→型宣言+nullable明示

## 7. ディレクトリ構造
（プロジェクトごとに記載。「触ってよい場所」と「触るな」を明示）
- src/main/... : 変更可
- generated/, vendor/, node_modules/ : 変更禁止
- 設定ファイル(build.gradle等) : 提案のみ、直接変更禁止

## 8. 報告フォーマット（毎回この形式で終える）
### 変更ファイル
- path/to/File.java : 変更理由（1行）
### 検証結果
- build: PASS/FAIL（コマンド出力の要約）
- test: PASS/FAIL
### 未解決・懸念
- なし / あれば列挙

## 9. エスカレーション条件（停止して質問）
- 仕様が2通り以上に解釈できる
- 破壊的変更・依存追加が必要
- 検証3回失敗
- セキュリティ・課金・データ削除に関わる変更
